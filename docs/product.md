# みんなで作る単語帳アプリ 仕様書（MVP）

## 1. 目的
- ログイン不要で、誰でも単語帳を作成・公開・閲覧し、クイズ形式で学習できるWebアプリを提供する。
- 学習体験は「問題文表示 -> 複数選択肢から回答」のシンプルな形式に統一する。

## 2. 前提技術
- フロントエンド/バックエンド: Next.js（App Router、`src` ディレクトリ構成）
- DB: PostgreSQL
- ORM: Prisma
- 認証: なし（匿名投稿）

## 3. 用語定義
- 単語帳: 複数の問題をまとめた公開コンテンツ
- 問題: 問題文と複数選択肢、正解を持つ1問
- 選択肢: 問題に紐づく回答候補

## 4. 想定ユーザー
- 単語帳を作りたいユーザー（投稿者）
- 単語帳で学習したいユーザー（回答者）

## 5. 画面一覧と機能要件

### 5.1 トップページ
**目的**
- 公開中の単語帳を一覧表示し、問題ページへの導線を提供する。

**要件**
- 公開済み単語帳のみ表示する。
- 各カードに以下を表示する。
  - 単語帳タイトル
  - 問題数
  - 作成日時（または更新日時）
- カードクリックで問題ページへ遷移する。
- 一覧は新着順（`publishedAt` 降順）とする。

### 5.2 検索ページ
**目的**
- タイトルで単語帳を検索し、目的の単語帳を見つけやすくする。

**要件**
- ヘッダーの検索ボックスから遷移可能。
- タイトル部分一致検索（大文字小文字は区別しない）。
- 検索対象は公開済み単語帳のみ。
- 検索結果0件時は専用メッセージを表示する。

### 5.3 問題ページ
**目的**
- 単語帳に含まれる問題を順番に出題し、回答できる。

**要件**
- 1問ずつ表示する。
- 問題文は英語/日本語どちらでも入力可能（テキストとして扱う）。
- 選択肢は複数表示し、ユーザーは1つ選択できる（単一選択）。
- 回答後に正誤を表示する。
- 「次へ」ボタンで次の問題へ進む。
- 最終問題終了後、結果画面（正答数/総問題数）を表示する。

### 5.4 単語帳作成モーダル
**目的**
- ヘッダーから即座に単語帳作成を開始する。

**要件**
- ヘッダーに「単語帳を作成」ボタンを配置。
- クリックでモーダル表示。
- 入力項目: 単語帳タイトル（必須）
- 「作成して問題編集へ」実行時に下書き単語帳を作成し、問題作成ページへ遷移する。

### 5.5 問題作成ページ
**目的**
- 単語帳に紐づく問題を追加・編集し、公開する。

**要件**
- 初期表示時に問題カードを1つ表示する。
- 問題カードには以下を持つ。
  - 問題文（必須）
  - 選択肢入力欄（最小2件）
  - 正解選択（1つ）
- 「問題追加」ボタンで問題カードを追加できる。
- 「公開」ボタンでバリデーション後にDB保存し、単語帳を公開状態にする。

## 6. バリデーション仕様

### 6.1 単語帳
- タイトル: 必須、1〜100文字

### 6.2 問題
- 問題文: 必須、1〜300文字
- 選択肢数: 2〜6件
- 各選択肢テキスト: 必須、1〜100文字
- 正解: 必ず1つ選択されていること

### 6.3 公開時
- 問題が1件以上存在すること
- 全問題が要件を満たすこと
- 1問でも不備がある場合は公開不可

## 7. データモデル（Prisma想定）

```prisma
model Deck {
  id          String     @id @default(cuid())
  title       String
  isPublished Boolean    @default(false)
  publishedAt DateTime?
  createdAt   DateTime   @default(now())
  updatedAt   DateTime   @updatedAt
  questions   Question[]
}

model Question {
  id         String   @id @default(cuid())
  deckId     String
  prompt     String
  orderIndex Int
  createdAt  DateTime @default(now())
  updatedAt  DateTime @updatedAt

  deck       Deck     @relation(fields: [deckId], references: [id], onDelete: Cascade)
  choices    Choice[]
}

model Choice {
  id          String   @id @default(cuid())
  questionId  String
  text        String
  isCorrect   Boolean  @default(false)
  orderIndex  Int
  createdAt   DateTime @default(now())
  updatedAt   DateTime @updatedAt

  question    Question @relation(fields: [questionId], references: [id], onDelete: Cascade)
}
```

## 8. 主要フロー

### 8.1 単語帳作成〜公開
1. ユーザーがヘッダーの作成ボタンを押す
2. モーダルでタイトル入力し作成
3. 下書き単語帳作成後、問題作成ページへ遷移
4. 問題を1問以上作成
5. 公開ボタン押下
6. バリデーション成功時 `isPublished=true`, `publishedAt=now()` を保存
7. 問題ページまたはトップへ遷移

### 8.2 回答フロー
1. トップ/検索から単語帳を開く
2. 1問目を表示
3. 選択肢を選んで回答
4. 正誤表示
5. 全問題終了で結果表示

## 9. URL設計（App Router想定）
- `/` : トップページ（単語帳一覧）
- `/search` : 検索ページ
- `/decks/[deckId]` : 問題ページ
- `/decks/[deckId]/edit` : 問題作成ページ

## 10. API/サーバー処理方針（MVP）
- 一覧取得: 公開単語帳を取得（ページング対応可能な形）
- 検索: タイトル部分一致検索
- 下書き作成: タイトルでDeck作成
- 問題保存: Question/Choiceをトランザクションで保存
- 公開: 公開前バリデーション後に公開状態へ更新
- 回答採点: MVPではクライアント側採点可（正解データはサーバー取得）

## 11. 非機能要件（MVP）
- パフォーマンス: 一覧・検索の初回表示は体感1秒台を目標
- 可用性: 投稿失敗時は再試行可能なエラー表示を行う
- セキュリティ:
  - XSS対策（入力値を適切にエスケープ）
  - レート制限（匿名投稿の乱用対策、将来拡張）
  - 不正な公開リクエストに対するサーバー側バリデーション

## 12. 制約・注意事項
- ログイン機能がないため、投稿者による後編集・削除はMVPでは対象外。
- 匿名投稿ゆえ、悪意ある投稿対策（NGワード、通報、rate limit）は将来優先で検討。

## 13. MVPの完了条件
- トップで公開単語帳一覧が見られる
- タイトル検索ができる
- 問題を作成して公開できる
- 問題ページで回答し、正誤と最終結果が表示される
